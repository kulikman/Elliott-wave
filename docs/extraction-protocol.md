## ЧАСТЬ 1 — ПРИНЦИПЫ (не нарушать)

### Принцип 1: Один источник за раз
В контекст Claude никогда не попадает больше одной книги.
Смешивание Нили и Уильямса в одном сессии → конфликтующие AKU
без явных пометок → отравление базы.

### Принцип 2: Один тип AKU за проход
Каждый проход (один запрос Claude) = один тип:
- Проход A: только definitions
- Проход B: только mandatory rules
- Проход C: только conditional rules
- Проход D: только heuristics
- Проход E: только exceptions и patterns

Смешивание типов → просадка точности после ~5 AKU.

### Принцип 3: Лимит на один ответ
Максимум 10 AKU за один запрос.
При большем количестве — качество падает нелинейно.
Если глава большая → разбивать на секции.

### Принцип 4: Сначала карта, потом извлечение
Перед AKU-извлечением из главы — сначала отдельный запрос:
"Составь карту этой главы: список секций с кратким описанием содержимого."
По карте — отдельный запрос на каждую секцию.

### Принцип 5: Цитата обязательна
Каждый AKU должен содержать verbatim_quote из источника.
Нет цитаты → status: draft, requires_review: true, НЕ verified.

### Принцип 6: Формализация — отдельный проход
НИКОГДА не делать в одном запросе:
❌ "Извлеки правило И формализуй его в псевдокоде"
✅ Сначала извлечь content. Потом (отдельный день) — формализовать.

### Принцип 7: Сомнение = requires_review: true
Если Claude не уверен в трактовке — выставить флаг.
Это честно и полезно. Лучше флаг чем уверенная неправда.

## ЧАСТЬ 2 — ПРОЦЕДУРА РАБОТЫ С ГЛАВОЙ

```
ШАГ 1: КАРТА ГЛАВЫ
  Запрос: "Это глава X книги Нили. Составь карту:
           список всех секций с кратким описанием содержимого.
           Формат: номер | название | содержимое (1-2 предложения)"
  Ожидаемый выход: таблица 5-15 строк
  Проверь: все секции покрыты?

ШАГ 2: КЛАССИФИКАЦИЯ СЕКЦИЙ
  По карте — отметь для каждой секции:
  [ ] содержит определения
  [ ] содержит mandatory rules
  [ ] содержит heuristics
  [ ] содержит паттерны (фигуры)
  [ ] содержит примеры
  [ ] требует Vision (есть диаграммы)

ШАГ 3: ПРОГОНЫ ПО ТИПАМ
  Для каждого типа — отдельный запрос.
  Передавать в контекст ТОЛЬКО текущую секцию + промпт.

ШАГ 4: VISION ДЛЯ ДИАГРАММ
  Отдельный проход — только картинки из PDF.
  Промпт: специальный (см. ниже).

ШАГ 5: ВЕРИФИКАЦИЯ
  Человек (Anton) читает AKU и:
  - проверяет verbatim_quote на соответствие тексту
  - меняет status: draft → verified или disputed
  - заполняет review_notes где нужно

ШАГ 6: ФОРМАЛИЗАЦИЯ
  Отдельный проход — только для verified mandatory rules.
  Claude видит: AKU без формализации + текст секции.
  Задача: только constraint в псевдокоде.
```

## ЧАСТЬ 3 — ПРОМПТЫ

### Промпт для извлечения DEFINITIONS

```
Ты извлекаешь определения из книги по волновому анализу.

КНИГА: {book_title}
ГЛАВА: {chapter_name}
СЕКЦИЯ: {section_name}
СТРАНИЦЫ: {page_range}

ПРАВИЛА РАБОТЫ:
1. Извлекай ТОЛЬКО явные определения терминов
2. Каждое определение = один AKU
3. Включай дословную цитату из текста
4. НЕ интерпретируй — только передавай
5. Если термин определяется несколько раз — создай несколько AKU
6. Максимум 10 AKU за один ответ

ДОСТУПНЫЕ ТОПИКИ: {taxonomy_list}

СХЕМА AKU: {schema_summary}

ПРИМЕР ХОРОШЕГО AKU: {golden_example}

ТЕКСТ СЕКЦИИ:
---
{section_text}
---

Выведи YAML-список AKU. Только YAML, никакого текста вокруг.
id начинай с {next_id}.
```

### Промпт для извлечения MANDATORY RULES

```
Ты извлекаешь обязательные правила из книги по волновому анализу.

КНИГА: {book_title}
ГЛАВА: {chapter_name}
СЕКЦИЯ: {section_name}

ПРАВИЛА РАБОТЫ:
1. Mandatory rule = нарушение которого делает волновой счёт невозможным
2. Если автор использует слова: "никогда", "не может", "обязательно",
   "невозможно", "всегда" — это кандидат на mandatory
3. Если есть сомнение mandatory vs heuristic → strength: controversial,
   requires_review: true
4. Включай дословную цитату
5. НЕ формализуй — только content
6. Максимум 10 AKU

ДОСТУПНЫЕ ТОПИКИ: {taxonomy_list}
ПРИМЕР: {golden_aku_0001}

ТЕКСТ СЕКЦИИ:
---
{section_text}
---

Выведи YAML-список. Только YAML.
```

### Промпт для Vision (диаграммы)

```
Это страница из книги по волновому анализу Эллиота (метод Нили/NeoWave).

КОНТЕКСТ: Глава {chapter}, страница {page}, книга {book_title}.

Если на изображении есть волновая разметка:
- Опиши точно какие волны обозначены (1,2,3,4,5 или A,B,C)
- Укажи направление каждой волны (вверх/вниз)
- Опиши соотношения длин если подписаны (числа, проценты)
- Воспроизведи все текстовые подписи дословно
- Опиши структурные обозначения если есть (:3, :5, :F3 и т.д.)

Если есть таблица:
- Воспроизведи в markdown-таблице со всеми данными

Если это блок-схема или алгоритм:
- Опиши все узлы и стрелки

Сохраняй все числа, проценты, аббревиатуры точно как в оригинале.
Если изображение содержит только текст — выведи текст.
```

## ЧАСТЬ 4 — КРАСНЫЕ ФЛАГИ (признаки галлюцинации)

При ревью AKU — немедленно ставь status: disputed если:

- В verbatim_quote есть слова которых нет в оригинале
- AKU содержит числа (проценты, коэффициенты) без цитаты
- statement формулирует то чего в тексте нет явно ("следовательно...", "из этого вытекает...")
- formalization.constraint содержит операторы которых нет в statement
- Один AKU ссылается на правила из разных глав/тем одновременно
- topic не соответствует statement_ru

## ЧАСТЬ 5 — ПРИОРИТЕТ ОБРАБОТКИ ГЛАВ (Нили)

Рекомендуемый порядок для первой книги:

```
1. Глава 2 — Моноволны          (фундамент, нужен для всего)
2. Глава 3 — Предварительный анализ   (правила длин, условия)
3. Глава 5 — Импульсные паттерны      (первый приоритет для индикатора)
4. Глава 6 — Корректирующие паттерны  (зигзаги, плоские)
5. Глава 4 — Дальнейшие построения    (группировка)
6. Глава 7 — Порядок волн             (степени)
7. Главы 8-12 — Продвинутые разделы   (второй приоритет)
```

Не начинать следующую главу пока текущая не прошла Gate:
- Все AKU из главы созданы
- Validator зелёный
- Антон верифицировал ≥80% AKU
