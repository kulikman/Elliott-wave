# Elliott Wave Brain — Полное описание проекта

## Что это и зачем

**Elliott Wave Brain** — система автоматического распознавания волновых паттернов Эллиота по методологии **Glenn Neely** (книга «Mastering Elliott Wave») с последующим бэктестом торговой стратегии.

**Конечная цель в двух уровнях:**
1. **Pine Script индикатор** для TradingView — автоматически размечает волны на графике, распознаёт паттерны (Импульс / Зигзаг / Плоская / Треугольник / Двойная Коррекция), подтверждает их по правилам Нили.
2. **Python ML pipeline** — на основе распознанных паттернов выдаёт торговые сигналы с предсказанием следующего движения (направление + амплитуда).

---

## Источник правил: AKU-система

Из книги Нили извлечено **247 AKU** (Atomic Knowledge Units) — атомарных правил волнового анализа с прямыми цитатами из книги. Они хранятся в `aku/` и формализованы в **spec_v3.0.json** (246 правил в 5 секциях). Каждое правило трассируемо до конкретного AKU.

---

## Pine Script: текущий статус

Два файла в `pine/`:

### `ewb_monowaves_mtf.pine` — Уровни 1-4
- **Уровень 1:** детектор моноволн (ZigZag по ATR). Моноволна = движение между двумя сменами направления по Нили (Гл.2).
- **Уровень 2:** Определитель Правил П1-7 + Условие a-f (AKU-0141/0142, Гл.3). Каждая моноволна классифицируется как `:5` (импульсная компонента) или `:F3` (коррективная).
- **Уровень 3:** Распознавание фигур по Структурным Сериям (Гл.4). Паттерны: Импульс (5-3-5-3-5), Зигзаг, Плоская, Треугольник, Двойная Коррекция.
- **Уровень 4:** Inline-подтверждение найденных фигур по правилам Нили + каналы Нили + Fibonacci цели.
- **MTF:** параллельный детектор на старшем ТФ (пунктирные линии). **Важно:** HTF bias — это информация "на будущее" и в Pine до сих пор содержит look-ahead баг (пивот рисуется на баре экстремума, а не на баре подтверждения разворота).

### `ewb_confirm.pine` — Уровень 4 (ручной)
Отдельный инструмент: пользователь вводит bar-offset 5 пивотов вручную, индикатор проверяет все правила Нили (AKU-0001..0009, AKU-0126/0127 и др.) и показывает таблицу ошибок/предупреждений.

**Известный баг в обоих файлах:** используется rule-based base sequence matcher (`:5`/`:F3` → `[5,3,5,3,5]`). На реальных данных он не находит импульсы, потому что коррективные волны внутри импульса дают Правило 6-7 → `?` вместо `:F3`. **Фикс разработан в Python (гибридный matcher), но ещё не перенесён в Pine.**

## Python Pipeline: текущий статус

Директория `python/ewb/`:

| Файл | Что делает |
|---|---|
| `monowaves.py` | ZigZag ATR детектор, зеркало Pine. Содержит `Pivot.confirmation_idx` — бар когда разворот подтверждён (критично против look-ahead). |
| `rules.py` | Определитель Правил П1-7 + Условие a-f + `rule_to_structure()` |
| `figures.py` | **Гибридный matcher** — проверяет чередование направлений + правила подтверждения напрямую. Находит Impulse/Triangle/Flat/Zigzag/DoubleCorr. |
| `confirm.py` | Все правила подтверждения (AKU-0001..0009, AKU-0017/0018/0022/0025/0126/0127/0203) |
| `htf.py` | HTF bias series — использует `confirmation_idx` пивотов (не `idx` экстремума) |

### Найденные и исправленные look-ahead баги

В ходе работы обнаружены **два критических look-ahead bias** которые давали ложные результаты:

1. **entry на экстремуме** (`end_idx`): первоначально entry брался на баре пивота-экстремума. Но этот бар невозможно использовать в реальности — мы не знали что это пивот пока разворот не подтверждён. **Фикс:** `entry_idx = pivot.confirmation_idx` (бар где цена пробила ATR×2.5 от экстремума).

2. **HTF пивот timing**: `htf_bias_series` первоначально использовал `htf_df.index[p.idx]` (экстремум HTF пивота) как момент когда он "стал известен". Тот же look-ahead. **Фикс:** `htf_df.index[p.confirmation_idx]`.

До фикса: видимый hit rate 77-82% (иллюзия). После: 50-63% (реально).

## Датасет и результаты исследования

### Датасет (`python/data/`)

- `figures_wide.parquet` — **10 397 фигур**, 58 тикеров × {1d, 1h}, 5 лет/730 дней
- `figures_all_tfs.parquet` — **17 183 фигуры**, добавлены 4h / 30m / 15m / 1w
- Features на каждую фигуру: amp_pct, w1..w4 ratios, duration, htf_bias, with_htf, future_ret_5/10/20/50/100

### Ключевые выводы (достоверные, после фиксов)

| Паттерн | Edge? | Примечание |
|---|---|---|
| **Flat fade** | ✅ Sharpe 1.80 | mean reversion после плоской коррекции |
| **Double Correction fade** | ✅ Sharpe 3.08, win 95% (n=120) | сильнейший сигнал, мало сделок |
| Impulse fade | ❌ Sharpe −0.66 | тренды продолжаются |
| Triangle fade | ❌ Sharpe −2.83 | треугольники прорываются непредсказуемо |
| Triangle follow | ❌ Sharpe −2.71 | обе стороны в минус → чистый шум |
| HTF bias filter | ⚠️ Нейтрален | до фикса казался полезным (был look-ahead) |

**Вывод по треугольнику:** fade −2.83, follow −2.71 — обе стороны убыточны. Это значит не "шум симметричный нулю", а что цена после треугольника ведёт себя случайно И комиссии доедают. Треугольник — структура накопления энергии перед неизвестным прорывом, без предсказуемого направления.

## Что ещё не сделано (приоритизировано)

### 🔴 Критично для production

**1. Перенести гибридный matcher в Pine**
Текущий Pine (`ewb_monowaves_mtf.pine`) использует старый rule-based matcher → 0 импульсов на реальных данных. Python версия с гибридным matcher работает корректно. Нужно перенести логику `_directions_alternate()` + `_try_impulse()` и т.д. в Pine Script v6.

**2. Убрать look-ahead из Pine**
В Pine `bar_index` пивота = бар экстремума. Нужно добавить задержку N баров (confirmation lag ≈ ATR×2.5 от экстремума) — примерно как это сделано в Python через `confirmation_idx`.

### 🟡 Важно для улучшения

**3. Sprint 5 — ML фильтр**
LightGBM на 1061 сделке: features = amp_pct, w-ratios, duration, confirmation_lag, vol_regime. Цель: отфильтровать bottom-50% сделок. Если поднимет Sharpe до 4+, добавить как опциональный фильтр.

**4. Backtest на 4h/30m/15m** (начат но не завершён)
Файл `python/scripts/backtest_all_tfs.py` написан, не запущен. Стратегия на 1h даёт Sharpe 3.08 — интересно проверить 4h (меньше сделок, крупнее движения) и 15m/30m (больше сделок, выше частота).

**5. Robustness check**
Добавить 50+ mid-cap тикеров (Russell 2000), период 2018-2021 (если доступен через yfinance), стресс-тест COVID 2020.

### 🟢 Production

**6. Telegram Bot / FastAPI**
Daily cron: сканировать 58 тикеров, находить свежие Flat/DC паттерны, отправлять алерт с ticker + TF + direction + entry zone + TP + SL.
